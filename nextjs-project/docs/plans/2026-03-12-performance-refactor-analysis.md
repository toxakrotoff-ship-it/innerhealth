# Анализ производительности и план рефакторинга

**Дата:** 2026-03-12  
**Цель:** Подготовка к рефакторингу с упором на производительность сайта, скорость загрузки и оптимизацию кода (LCP, размер бандла, TTFB, архитектура, Core Web Vitals).

**Контекст:** Next.js 16 (App Router), React 19, Tailwind 4, Prisma 7. Существующие заметки по рефакторингу — `src/services/refactoring_plans.md` (Sprints 3–5 касаются производительности).

---

## 1. Текущие риски и узкие места

### 1.1 TTFB и количество запросов к БД (главная страница)

- **Layout `(site)/layout.tsx`:**
  - На каждый запрос выполняется `getSettingsMap()` — один запрос к БД для настроек.
  - `export const dynamic = 'force-dynamic'` отключает статическую отдачу всего дерева (site); каждый запрос идёт на сервер. В сочетании с `revalidate = 60` поведение может быть неочевидным (см. документацию Next.js).
- **Главная страница `(site)/page.tsx`:**
  - `getHomeData()`: один `Promise.all` с 5 запросами (categories, newProducts, newsPosts, articlePosts, reviews) — хорошо.
  - Отдельный `Promise.all` с **6 вызовами `getResolvedBlock()`**. Каждый `getResolvedBlock(page, key)` внутри вызывает `getResolvedBlocksForPage(page)`. Итого:
    - 5 раз `getResolvedBlocksForPage('home')` и 1 раз `getResolvedBlocksForPage('catalog')` — **6 лишних round-trip к БД** вместо двух (один для `home`, один для `catalog`) с последующим выбором блоков в памяти.
  - **HeroBlock:** 4 вызова `getResolvedBlock('home', …)` → ещё **4 вызова `getResolvedBlocksForPage('home')`**.
- **Итого по главной:** минимум 5 (getHomeData) + 6 (блоки на странице) + 4 (Hero) = **15 обращений к БД**, при этом `getResolvedBlocksForPage('home')` вызывается **9 раз**. Это основной кандидат на снижение TTFB.

**Рекомендация:** Ввести батчинг контент-блоков: один вызов `getResolvedBlocksForPage('home')` и один `getResolvedBlocksForPage('catalog')` на рендер главной, передавать результат в HeroBlock и в страницу (или кешировать на время запроса). Либо добавить `getResolvedBlocksForPages(pageIds: string[])` и выбирать блоки по ключам в JS.

### 1.2 Изображения и LCP

- **Hero:** `hero-block.tsx` использует `next/image` с `priority` и `sizes` — хорошо. Картинка `/hero-portrait.png` — локальный статик; убедиться, что файл оптимизирован (WebP/AVIF уже в `next.config`: `formats: ['image/avif', 'image/webp']`).
- **Каталог/карточки:** В `product.service.ts` для списков используется `productCardSelect` с полем `photos` (полный JSON). Для карточек достаточно первого фото и blur placeholder; тянуть весь массив фото увеличивает объём данных и память.
- **ProductCard:** клиентский компонент (`'use client'`), с `next/image`, `priority` и `blurDataURL` для первых двух товаров — разумно. Стоит проверить, что `sizes` везде заданы адекватно сетке (например, для 4 колонок — порядка `25vw` на десктопе).
- **Дубликаты файлов:** В репозитории есть пары вида `adaptive-nav.tsx` / `adaptive-nav 2.tsx`, `fluid-grid.tsx` / `fluid-grid 2.tsx`, `responsive-text.tsx` / `responsive-text 2.tsx`, `scalable-spacing.tsx` / `scalable-spacing 2.tsx`. Нужно оставить один вариант, обновить импорты и удалить дубликаты, чтобы не раздувать бандл и не путать сборку.

### 1.3 Бандл и клиентские компоненты

- **Динамические импорты на главной:** SprintPowerBlock, PartnersBlock, ReviewsCarousel, ReviewCtaBlock подключаются через `dynamic(..., { ssr: true })`. Это уменьшает размер начального JS, но все они всё равно рендерятся на сервере; при необходимости можно рассмотреть `ssr: false` для некритичных блоков (например, партнёры/отзывы ниже первого экрана), чтобы ещё уменьшить клиентский бандл.
- **Layout:** CartDrawer и CookieConsent загружаются через `nextDynamic` без SSR — хорошо для первого рендера.
- **Много `'use client'`:** В `src/components/site/` и страницах много клиентских компонентов. Часть из них (например, чисто презентационные с пропсами) можно перевести в RSC, чтобы уменьшить клиентский JS. В `refactoring_plans.md` (Sprint 3) уже заложено: максимизировать RSC, оставляя `'use client'` только там, где есть интерактивность или браузерные API.
- **Zustand/cart:** Проверить, что в хранилище корзины (и при персисте в localStorage) не сохраняются тяжёлые объекты (все поля продукта), а только минимальный набор (id, quantity, slug и т.п.) — как в Sprint 3.

### 1.4 Кеширование и заголовки

- **Каталог:** В `next.config.ts` для `/catalog` и `/catalog/:path*` задан `Cache-Control: public, max-age=3600, stale-while-revalidate=3600` — хорошо.
- **Главная:** При `dynamic = 'force-dynamic'` страница не кешируется на edge/CDN как статическая; каждый запрос идёт на сервер. Если допустимо отдавать главную с задержкой до 60 сек, можно рассмотреть отказ от `force-dynamic` только для этой маршрутной группы и использование `revalidate` (с учётом ограничения по доступу к БД при сборке в Docker — возможно, только runtime revalidate).

### 1.5 Код и архитектура

- **Контент-блоки:** Логика «один ключ — один вызов getResolvedBlock» приводит к N вызовам `getResolvedBlocksForPage` для одной страницы. Централизовать загрузку блоков на уровень страницы/лейаута (один запрос на страницу или на набор страниц).
- **DAL:** В проекте уже есть сервисы (`product.service`, `content-block.service`, `settings.service`). Прямые вызовы `prisma` в `app/(site)/**` (например, в `page.tsx`, `catalog/[categorySlug]/page.tsx`, `news/[slug]/page.tsx`) лучше постепенно сводить к вызовам сервисов, чтобы единообразно применять `select`, кеш и индексы.
- **Типы:** Избегать `any`; использовать общие DTO (например, `ProductCardDTO`) и типы из `src/types` там, где они уже есть.
- **Главная страница:** В `page.tsx` используются компоненты `Heading2`, `SpacingVertical`, `FluidGrid` без видимых импортов в начале файла. Необходимо добавить явные импорты из `@/components/ui/responsive-text`, `@/components/ui/scalable-spacing`, `@/components/ui/fluid-grid`, чтобы избежать ошибок сборки/рантайма и корректно учитывать бандл.

### 1.6 Прочее

- **SiteHeader:** Выполняет `getServerSession(authOptions)` и при необходимости проверку cookies — это добавляет задержку к каждому запросу. Оставить как есть с точки зрения безопасности; при росте нагрузки рассмотреть кеширование результата сессии на очень короткий TTL или оптимизацию на стороне NextAuth.
- **Console.log в SiteHeader:** Удалить или заменить на условный лог (только в dev), чтобы не засорять прод и не тратить время на вывод.
- **Индексы Prisma:** По `refactoring_plans.md` (Sprint 4) стоит проверить индексы по полям `slug`, `status`, `published`, `page` (ContentBlock) и другим часто используемым в фильтрах.

## 2. Приоритизированный чек-лист рефакторинга

Сделать в первую очередь (максимальный эффект при разумных трудозатратах):

1. **Батчинг контент-блоков на главной**
   - Загружать блоки страниц `home` и `catalog` одним вызовом на страницу (или двумя: home + catalog).
   - Передавать результат в `HeroBlock` и в `HomePage` вместо 6+4 отдельных вызовов `getResolvedBlock`.
   - При необходимости ввести `getResolvedBlocksForPages(pages: string[])` или кеш на время одного request (например, в `content-block.service`).

2. **Исправление импортов на главной**
   - Добавить в `(site)/page.tsx` импорты `Heading2`, `SpacingVertical`, `FluidGrid` из соответствующих UI-модулей (и при необходимости убрать дубликаты компонентов " 2" после консолидации).

3. **Удаление дубликатов компонентов**
   - Удалить файлы вида `adaptive-nav 2.tsx`, `fluid-grid 2.tsx`, `responsive-text 2.tsx`, `scalable-spacing 2.tsx`, `adaptive-demo 2.tsx` и перевести все импорты на основные файлы.

4. **Оптимизация выборки продуктов для списков**
   - Для каталога и главной не тянуть полный `photos` JSON; в сервисе выбирать только первое фото или отдельное поле с blur/thumbnail, если оно есть в схеме.

5. **Снижение водопада запросов на главной**
   - Объединить второй `Promise.all` (6 блоков) с данными Hero: один раз загрузить `getResolvedBlocksForPage('home')` и `getResolvedBlocksForPage('catalog')`, затем распределить блоки по компонентам.

Далее (средний приоритет):

6. Пересмотреть использование `force-dynamic` в `(site)/layout.tsx`: возможно, вынести в отдельный route group только страницы, которым нужна строгая динамика, или документировать причину и оставить как есть.
7. Добавить/проверить индексы в Prisma по полям, используемым в частых запросах (slug, status, published, page и т.д.).
8. Аудит `'use client'`: перевести в RSC компоненты без интерактивности и браузерных API (см. Sprint 3 в refactoring_plans.md).
9. Проверить Zustand persist: сохранять в localStorage только минимальный набор полей для корзины.
10. Убрать или обернуть в `if (process.env.NODE_ENV === 'development')` логи в SiteHeader.

Низкий приоритет / по мере доработок:

11. Рассмотреть `ssr: false` для части динамических блоков ниже первого экрана (PartnersBlock, ReviewCtaBlock), если метрики покажут выигрыш по FCP/JS.
12. Единый формат DTO для карточек товаров и типы в `src/types`.
13. Документировать решение по `dynamic = 'force-dynamic'` и кешированию в README или в этом плане.

## 3. Связь с существующими планами

- **Sprint 3 (refactoring_plans.md):** RSC, динамические импорты, Zustand, перенос фетчинга на сервер — напрямую поддерживает цели по бандлу и TTFB.
- **Sprint 4:** Выборка только нужных полей, индексы, типы — снижает нагрузку на БД и память, улучшает TTFB.
- **Sprint 5:** Локальные изображения, sharp при загрузке, WebP/AVIF, `sizes`/`priority`/`placeholder="blur"` — напрямую влияет на LCP и размер ответов.

Текущий анализ дополняет эти спринты конкретными точками (главная страница, контент-блоки, дубликаты файлов, импорты) и приоритетами для следующего этапа работ.

## 4. Метрики для проверки после рефакторинга

- **LCP:** главная и страница товара (hero и первый экран).
- **TTFB / FCP:** главная с холодным запуском и при повторных запросах.
- **CLS:** проверка после изменений в изображениях и сетках.
- **Размер JS:** сравнение до/после (например, через `next build` и анализ чанков или `@next/bundle-analyzer`).
- Количество запросов к БД на один рендер главной (целевой ориентир: снизить с ~15 до 7–8 за счёт батчинга блоков и при необходимости объединения других запросов).

После согласования этого анализа можно перейти к пошаговому плану реализации (отдельный документ или задачи в трекере).

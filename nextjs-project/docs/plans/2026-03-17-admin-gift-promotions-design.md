# Подарочные акции (подарок в заказ) — Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Реализовать гибкую механику подарочных акций (товар в подарок с ценой 0 ₽) с конфигурацией в админке и отображением на сайте.

**Architecture:** Подарочные акции хранятся в отдельной Prisma‑модели, применяется на сервере при создании заказа в `orderService`, фронт помечает подарочные строки и использует существующий редактор текстов для описаний акций. Страница «Акции» на сайте читает эти же сущности и рендерит информационные карточки без кнопок покупки.

**Tech Stack:** Next.js App Router, React, TypeScript, Prisma, PostgreSQL, Zod, существующий rich‑text редактор в админке.

---

## 1. Цели и общая идея

Фича позволяет в админке настраивать **подарочные акции**, при которых в заказ автоматически добавляется **подарочный товар с ценой 0 ₽** при выполнении заданных условий.

Основные цели:
- **Маркетинг**: стимулировать покупки через гибко настраиваемые подарки.
- **Гибкость**: дать максимум вариантов конфигурации (условия, количество, взаимодействие с акциями и промокодами).
- **Прозрачность**: понятная модель для заказчика в админке и предсказуемое поведение для пользователя на сайте.
- **Расширяемость**: чтобы на этой схеме можно было дальше строить более сложные промо‑механики.

---

## 2. Сущности и терминология

- **Подарочная акция** — конфигурация, описывающая:
  - какой товар является **подарком**;
  - при каких **условиях** он добавляется в заказ;
  - как считается **количество подарков**;
  - как подарок ведёт себя совместно с другими скидками;
  - как акция отображается на сайте в разделе «Акции».

- **Подарочный товар** — обычный товар из каталога, который при срабатывании акции добавляется в заказ с ценой **0 ₽**.

- **Триггер‑товар** — товар, покупка которого (в нужном количестве) даёт право на подарок.

- **Условие по сумме корзины** — минимальная сумма корзины, после достижения которой подарок должен быть добавлен.

- **Срабатывание условия** — факт выполнения условия для данной корзины (заказа). Например, при условии «каждые 2000 ₽ в корзине» и сумме 4500 ₽ количество срабатываний = 2.

- **Режим количества подарков**:
  - «1 подарок на заказ»
  - «Подарков по количеству срабатываний условия».

## 3. Конфигурация подарочной акции в админке

### 3.1. Базовая информация

Для каждой подарочной акции настраиваем:

- **Название акции** (string, required)  
  Отображается в админке и может использоваться на сайте, если не задан отдельный заголовок.

- **Статус**:
  - `enabled` (вкл) / `disabled` (выкл).

- **Период действия**:
  - `validFrom` (datetime, nullable) — дата начала действия.
  - `validTo` (datetime, nullable) — дата окончания.
  - Логика:
    - если `validFrom` не задано — акция начинает действовать сразу после включения;
    - если `validTo` не задано — бессрочно;
    - при проверке учитываем, что `now` должен попадать в интервал `[validFrom, validTo]` (с учётом null).

- **Подарочный товар**:
  - `giftProductId` (string, required) — ссылка на модель товара.
  - Товар выбирается из существующего каталога (поиск по названию, SKU, ID).
  - При срабатывании акции в заказ добавляется строка с этим товаром:
    - `price = 0`
    - `quantity` зависит от настроек количества подарков.

- **Лимиты (опционально, можно реализовать в следующей итерации)**:
  - `maxGiftsPerOrder` (int, nullable) — максимум подарков по этой акции на один заказ.
  - `maxGiftsPerUser` (int, nullable) — максимум подарков по этой акции на одного пользователя (идентификация по `userId`, при его отсутствии можно не применять).
  - `globalGiftLimit` (int, nullable) — общий лимит подарков по этой акции на всю систему.

В первой реализации достаточно `maxGiftsPerOrder`, остальные можно либо сразу заложить в модель, либо добавить позже.

### 3.2. Тип условия срабатывания (trigger)

Тип условия — обязательное поле (radio в UI):

1. **Покупка конкретного товара**
   - Поля:
     - `triggerType = 'PRODUCT'`
     - `triggerProductId` (string, required) — ссылочный товар.
     - `triggerProductMinQty` (int, required, ≥ 1) — минимальное количество триггер‑товара.
   - Логика выполнения:
     - считаем суммарное количество строк в корзине с `productId = triggerProductId`;
     - если `totalQty >= triggerProductMinQty`, условие выполнено;
     - количество срабатываний (для режима «по количеству срабатываний») можно считать двумя способами:
       - **простая модель 1**: количество срабатываний = `floor(totalQty / triggerProductMinQty)`;
       - **простая модель 2**: если `totalQty >= triggerProductMinQty`, считаем как 1 срабатывание.  
     - Рекомендуемая логика для максимальной гибкости — **модель 1** (каждый полный «пакет» триггер‑товара даёт срабатывание).

2. **Сумма корзины ≥ X**
   - Поля:
     - `triggerType = 'CART_TOTAL'`
     - `minCartTotal` (decimal, required, ≥ 0) — минимальная сумма корзины.
     - `cartTotalCalculationMode` (enum) — как считать сумму:
       - `BEFORE_DISCOUNTS_EXCLUDING_GIFTS` — до скидок и промокодов, не включая подарочные позиции и товары с ценой 0.
       - (опционально на будущее: `AFTER_DISCOUNTS`, `BEFORE_DISCOUNTS_INCLUDING_PROMO_PRODUCTS` и т.п.)
   - Базовая логика на сейчас:
     - считаем сумму **только по обычным строкам заказа** (без подарков, без позиций с price=0);
     - можно считать до применения промокода, чтобы промокод не «ломал» выполнение условия подарка.
   - Количество срабатываний:
     - **Простая модель**: если `sum >= minCartTotal`, то есть **1 срабатывание**.
     - (Опционально в будущем) шаговый режим: каждое `minCartTotal` даёт ещё одно срабатывание, например каждые 2000 ₽ → ещё один подарок.

### 3.3. Режим количества подарков

В админке делаем переключатель:

- `giftQuantityMode` (enum):
  - `ONE_PER_ORDER` — **1 подарок на корзину**, если условие выполнено.
  - `PER_TRIGGER` — **подарков по количеству срабатываний условия**:
    - для `PRODUCT`:
      - количество срабатываний = `floor(totalQty(triggerProduct) / triggerProductMinQty)`;
      - итоговое количество подарков = `min(кол-во срабатываний, maxGiftsPerOrder ?? ∞)`.
    - для `CART_TOTAL` (в первой итерации):
      - если `sum >= minCartTotal` → считаем как 1 срабатывание (или на будущее — `floor(sum / minCartTotal)`).
      - итоговое количество подарков = `min(кол-во срабатываний, maxGiftsPerOrder ?? ∞)`.

### 3.4. Взаимодействие с акционными товарами и промокодами

Нужно учесть уже существующую механику:
- акционные товары (есть `priceOld` и/или `discountPrice`, флаги `hasPromoPrice`, `isPromoEligible` в корзине);
- промокоды, которые влияют только на eligible‑товары.

Для подарков добавляем в конфигурацию:

#### 3.4.1. Поведение при наличии акционных товаров

- Поле `promoProductInteractionMode` (enum):
  - `BLOCK_IF_PROMO_PRODUCTS_PRESENT` — **не выдавать подарок**, если в корзине есть акционные товары (hasPromoPrice).
  - `IGNORE_PROMO_PRODUCTS_IN_CONDITION` — при расчёте условия (сумма корзины) **игнорировать акционные товары**, но подарок всё равно можно выдать.
  - `ALWAYS_ALLOW` — подарки работают независимо от наличия акционных товаров.

В первой итерации можно реализовать минимум:
- `BLOCK_IF_PROMO_PRODUCTS_PRESENT`
- `ALWAYS_ALLOW`
и при необходимости добавить промежуточный режим позже.

#### 3.4.2. Совместимость с промокодами

- Поле `promoCodeInteractionMode` (enum):
  - `ALLOW_WITH_PROMOCODE` — подарок выдаётся даже при активном промокоде.
  - `BLOCK_IF_PROMOCODE_PRESENT` — если промокод применён, подарок **не выдаётся**, даже если условия выполнены.

Первая реализация: достаточно этих двух режимов.

### 3.5. Поведение при изменении корзины

Нужно определить, что происходит с подарком, если пользователь меняет корзину:

- Флаг `autoRemoveWhenConditionFails` (boolean):
  - Если `true`:
    - при любом пересчёте корзины/заказа, если условия акции перестали выполняться, **подарочные позиции по этой акции удаляются** или их количество уменьшается до нового рассчитанного.
  - Если `false`:
    - подарок, однажды добавленный, остаётся в заказе даже при изменении условий (потенциально более щедрый, но менее предсказуемый режим).
  - Рекомендуемый дефолт: `true`.

- Флаг `userCanRemoveGiftManually` (boolean):
  - Если `true`:
    - пользователь на фронте может удалить подарочную позицию (строка в корзине помечается как подарок, и на ней есть кнопка «Удалить»).
    - При следующем пересчёте корзины **не добавляем этот подарок обратно автоматически** в рамках той же сессии (можно хранить флаг в состоянии корзины).
  - Если `false`:
    - подарочную строку удалить нельзя (нет кнопки «Удалить»).

## 4. Отображение на сайте (раздел «Акции»)

Нужно, чтобы для каждой подарочной акции можно было вывести **информационную карточку** в публичном разделе «Акции».

### 4.1. Поля для сайта

В модель акции добавляем:

- `siteTitle` (string, nullable):
  - Заголовок для отображения на сайте.
  - Если не задан — используем `Название акции`.

- `siteDescription` (text / rich‑text / JSON — тот же формат, который уже используется для других редакторов):
  - Основное описание акции.
  - Редактируется тем же компонентом редактора, что и другие текстовые поля в админке (заказчику уже знакома модель).

- `showOnSite` (boolean):
  - Если `true` — акция показывается в разделе «Акции».
  - Если `false` — только backend‑логика, без публичного отображения.

### 4.2. Компонент на сайте

- Страница/раздел `/promotions` (или существующий раздел «Акции»):
  - Загружает список **активных** подарочных акций с `showOnSite = true`.
  - Фильтр по активности:
    - `status = enabled`
    - текущая дата в пределах `[validFrom, validTo]`.

- Карточка акции содержит:
  - Заголовок: `siteTitle` или `Название акции`.
  - Картинку:
    - можно подтягивать из `giftProduct.photo`.
  - Текст:
    - `siteDescription` рендерится через тот же рендерер, что и другие текстовые поля.
  - Автоматически сгенерированный блок «Условия» (опционально):
    - Например:
      - «Подарок: [название подарка] при покупке товара [название триггер‑товара] от [N шт]»
      - или «Подарок: [название подарка] при заказе от [X ₽]»
    - Этот блок можно добавить позже, сейчас достаточно редактора описания.

- Важно:
  - **Нет** кнопок «В корзину» и **нет** показов цен в карточке акции.
  - Это **информационный компонент**, не точка входа в корзину.

## 5. Поведение на фронте (корзина/оформление заказа)

### 5.1. Идентификация подарочных строк

Чтобы фронт понимал, какие строки в корзине являются подарками:

- В данных строки заказа/корзины (и в API) должен быть флаг, например:
  - `isGift: boolean`
  - `giftPromotionId: string | null` — если это подарок по конкретной акции.

Использование:
- При отображении строки с `isGift = true`:
  - показываем «Подарок» / «Подарок по акции»;
  - не даём менять цену;
  - опционально скрываем инпут количества, если не хотим позволять менять qty вручную.

### 5.2. Когда рассчитывать подарки

Варианты:
- а) На фронте (клиент считает правила самостоятельно).
- б) На бэкенде (API отдаёт уже рассчитанную корзину/заказ).

Рекомендуемая модель в рамках текущего проекта:
- **Бэкенд‑центричная логика**:
  - При создании заказа (`POST /api/orders`) backend:
    - грузит активные подарочные акции;
    - по переданным `items` считает, какие подарки нужно добавить;
    - модифицирует создаваемый `order` и `orderItems`, добавляя нужные подарочные позиции (price=0, isGift=true, giftPromotionId=...).
  - На фронте:
    - при оформлении заказа отправляем чистый состав корзины (без вручную добавленных подарков);
    - после ответа API показываем уже финальный заказ (с подарками).

Возможное улучшение в следующей итерации:
- отдельный эндпоинт предварительного расчёта (`POST /api/cart/preview`), который возвращает «корзину с подарками», чтобы в UI видно было подарок ещё до финального оформления.

В рамках данного дизайна важно именно **описать механику**, а не привязку к конкретным эндпоинтам — но сразу закладываем, что:
- подарки считаются **на сервере**;
- фронт доверяет серверу и отображает то, что пришло.

### 5.3. Порядок применения логики

Упрощённый порядок при создании заказа:

1. Валидация запроса (Zod‑схемы, уже есть).
2. Загрузка товаров по `productId` (как и сейчас).
3. Расчёт базовых сумм:
   - разделение на:
     - товары с промо‑ценой (hasPromoPrice),
     - товары eligible под промокод,
     - неучаствующие и т.п.
4. Применение промокода (если есть).
5. Загрузка всех **активных** подарочных акций.
6. Для каждой акции:
   - проверка периода `validFrom`/`validTo`;
   - проверка статуса `enabled`;
   - проверка `promoProductInteractionMode`:
     - при `BLOCK_IF_PROMO_PRODUCTS_PRESENT` — если в корзине есть акционный товар → пропускаем акцию;
   - проверка `promoCodeInteractionMode`:
     - при `BLOCK_IF_PROMOCODE_PRESENT` и наличии промокода → пропускаем акцию;
   - проверка условия:
     - `PRODUCT`:
       - считаем общее количество триггер‑товара;
       - если меньше `triggerProductMinQty` → нет срабатываний;
       - иначе считаем количество срабатываний согласно `giftQuantityMode`;
     - `CART_TOTAL`:
       - считаем сумму корзины по `cartTotalCalculationMode`;
       - если `sum < minCartTotal` → нет срабатываний;
       - иначе считаем количество срабатываний согласно `giftQuantityMode`.
   - учитываем `maxGiftsPerOrder` (и в будущем остальные лимиты).
   - если итоговое количество подарков > 0:
     - добавляем/корректируем строки заказа:
       - либо создаём новую строку с `giftProductId`, `price=0`, `quantity=calculated`, `isGift=true`, `giftPromotionId=...`;
       - либо, если подарок уже есть (повторный расчёт), синхронизируем количество.
7. Если `autoRemoveWhenConditionFails = true`:
   - для акций, где количество срабатываний стало 0, удаляем строки подарков.

## 6. UI в админке

### 6.1. Раздел и список акций

- Новый раздел в админке, например:
  - `Админка → Акции → Подарочные акции`.
- Таблица/список:
  - Название;
  - Статус (вкл/выкл);
  - Период действия (кратко);
  - Подарочный товар;
  - Тип условия (товар / сумма корзины);
  - Флаг «Показывать на сайте».

### 6.2. Форма создания/редактирования акции

Блоки формы:

1. **Основное**
   - Название акции (input).
   - Статус (switch).
   - Период действия (date‑pickers).

2. **Подарок**
   - Выбор подарочного товара (select/combobox по товарам).
   - Поле `maxGiftsPerOrder` (число, опционально).

3. **Условие**
   - Radio:
     - «Покупка конкретного товара»
     - «Сумма корзины ≥ X»
   - Если выбран **товар**:
     - Выбор триггер‑товара.
     - Минимальное количество триггер‑товара.
   - Если выбрана **сумма корзины**:
     - Минимальная сумма корзины.
     - (опционально) режим расчёта суммы, если решим дать его менять в UI.

4. **Количество подарков**
   - Переключатель:
     - «1 подарок на заказ»
     - «Подарков по количеству срабатываний условия».

5. **Взаимодействие с акциями и промокодами**
   - Поведение при наличии акционных товаров:
     - селект из доступных вариантов (минимум: «запретить подарок, если есть акционные товары» / «разрешить всегда»).
   - Совместимость с промокодами:
     - селект: «разрешить», «запретить, если есть промокод».

6. **Поведение подарка**
   - Авто‑удаление, если условия перестали выполняться (switch).
   - Разрешить пользователю удалить подарок вручную (switch).

7. **Отображение на сайте**
   - Показывать в разделе «Акции» (switch).
   - Заголовок для сайта (input).
   - Описание для сайта (текстовый редактор):
     - используем **ровно тот же компонент редактора**, что и для других описаний в админке, чтобы модель была знакома заказчику.

## 7. Границы первой итерации

Чтобы фича была реализуема поэтапно, для первой итерации можно зафиксировать:

- Поддерживаем:
  - Условия:
    - `PRODUCT` (с количеством).
    - `CART_TOTAL` (однократное срабатывание при достижении суммы).
  - Режимы количества подарков:
    - `ONE_PER_ORDER`.
    - `PER_TRIGGER` (для товара — `floor(qty / minQty)`; для суммы — пока 1 срабатывание).
  - Взаимодействие:
    - `promoProductInteractionMode`: минимум `BLOCK_IF_PROMO_PRODUCTS_PRESENT`, `ALWAYS_ALLOW`.
    - `promoCodeInteractionMode`: `ALLOW_WITH_PROMOCODE`, `BLOCK_IF_PROMOCODE_PRESENT`.
  - Поведение:

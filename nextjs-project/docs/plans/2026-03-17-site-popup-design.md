## Попап на главной — дизайн фичи

### 1. Цель и сценарий

- **Цель**: дать в админке полностью ручной редактор всплывающего попапа на главной странице для любых промо‑сообщений (акции, участие в выставках и т.п.).
- **Сценарий**: попап показывается **только на главной** (`/`), через заданное количество секунд после загрузки, с ограничением частоты показа и возможным автозакрытием.
- **Контент**: произвольный rich‑text + одна картинка (d&d загрузка) + опциональная кнопка/ссылка (в т.ч. можно вести в раздел «Акции»).

### 2. Модель данных

Добавляем новую сущность, условное имя `SitePopup` (одна запись, максимум один активный попап на сайт):

- **Идентификация и статус**
  - `id String @id @default(cuid())`
  - `title String` — внутреннее название попапа для админки.
  - `isEnabled Boolean @default(false)` — включён/выключен.

- **Контент**
  - `richJson Json?` — rich‑text содержимое в формате TipTap JSON (тот же формат, что и в `ContentBlock`).
  - `imageUrl String?` — опциональная картинка попапа. В админке — загрузка через drag & drop, на сервере сохраняем нормализованный путь.
  - `ctaLabel String?` — текст кнопки (например, «Подробнее», «Смотреть акции»).
  - `ctaUrl String?` — URL перехода (может быть относительный, типа `/catalog/aktsii`, или абсолютный).

- **Поведение**
  - `delaySeconds Int @default(5)` — через сколько секунд после загрузки главной показать попап.
  - `hideForDays Int @default(7)` — на сколько дней не показывать попап повторно пользователю после закрытия/показа (через localStorage/cookie).
  - `autoCloseSeconds Int?` — через сколько секунд автозакрывать попап (если `null` или `0` — не автозакрывать).

Технические поля:

- `createdAt DateTime @default(now())`
- `updatedAt DateTime @updatedAt`

Допущения:

- В первой итерации храним только один актуальный `SitePopup`. Если в будущем понадобится история/архив — можно оставить несколько записей и в коде брать последнюю включённую.
- Загрузка картинки реализуется как обычный upload (endpoint или встроенный механизм), в `imageUrl` сохраняем результат (относительный путь в `public/` или внешнюю ссылку).

### 3. Админский UI

Новый раздел в админке, например: `Админка → Контент → Попап на главной`.

#### 3.1. Общая структура страницы

- Одна страница с формой редактирования **единственного** попапа:
  - Без списка/таблицы — просто форма.
  - При первом заходе, если записи нет, создаём её автоматически с дефолтами.

#### 3.2. Поля формы

Блоки:

1. **Статус**
   - Switch `Включен` (`isEnabled`).

2. **Основное**
   - Поле `Название попапа` (`title`, input).

3. **Контент**
   - DnD‑зона для загрузки картинки:
     - Подсказка по рекомендуемым размерам/формату.
     - Превью загруженной картинки.
     - Возможность удалить/заменить картинку.
   - Rich‑text редактор (тот же компонент, что для `ContentBlock`/постов):
     - Заголовки, параграфы, списки, ссылки и т.д.
   - Поля кнопки:
     - `Текст кнопки` (`ctaLabel`, input).
     - `Ссылка` (`ctaUrl`, input).
     - Подсказка: «Можно указать `/catalog/aktsii`, чтобы вести в раздел акций».

4. **Поведение**
   - `Показать через N секунд` (`delaySeconds`, number input, min=0).
   - `Не показывать повторно N дней` (`hideForDays`, number input, min=0).
   - `Автозакрытие через M секунд` (`autoCloseSeconds`, number input, min=0; 0 или пусто = не автозакрывать).

5. **Превью**
   - Блок превью попапа, который:
     - рендерит тот же компонент, что и на фронте,
     - использует текущие значения формы (без ожидания сохранения).

#### 3.3. Валидация и UX

- Поля поведения:
  - `delaySeconds` и `hideForDays` должны быть неотрицательными целыми.
  - `autoCloseSeconds` либо пусто, либо неотрицательное целое.
- Если `ctaLabel` заполнен, но `ctaUrl` пуст — показываем предупреждение, но допускаем сохранение (кнопка без действия менее полезна, но не критична).
- При сохранении:
  - optimistic UI или стандартный loader.
  - уведомление об успехе/ошибке.

### 4. Поведение на сайте (главная страница)

#### 4.1. Где подключаем

- В layout/странице главной (например, `src/app/(site)/page.tsx`):
  - на сервере загружаем текущий активный `SitePopup`:
    - фильтр: `isEnabled = true`;
    - если в будущем будет несколько, берём последний по `updatedAt`.
  - пробрасываем данные в клиентский компонент `HomePopupClient`.

#### 4.2. Клиентский компонент

Клиентский компонент `HomePopupClient`:

- Пропсы:
  - объект с полями попапа (`richJson`, `imageUrl`, `ctaLabel`, `ctaUrl`, `delaySeconds`, `hideForDays`, `autoCloseSeconds`).
- Логика:
  1. При маунте:
     - если попап не передан — ничего не делаем.
     - читаем из `localStorage` ключ, например: `innerhealth_site_popup_<id>`.
     - если там дата последнего показа и она меньше `hideForDays` от текущей даты → **не показываем** попап.
  2. Если показывать можно:
     - ставим `setTimeout` на `delaySeconds * 1000`.
     - по таймеру открываем модалку.
     - если задан `autoCloseSeconds` > 0 — ставим второй таймер на автозакрытие.
  3. При закрытии попапа:
     - записываем в `localStorage` текущую дату (`new Date().toISOString()`).
     - очищаем таймеры.
  4. При клике по CTA‑кнопке:
     - по умолчанию просто `href`/`Link`, без спец‑логики.
     - можно дополнительно записывать дату показа в `localStorage` (как при закрытии).

#### 4.3. Компонент попапа (UI)

- Модальное окно на всю страницу:
  - полупрозрачный фон.
  - контентная карточка по центру (адаптивная, mobile‑first).
- Вёрстка:
  - Картинка (если есть) слева/сверху (адаптивно).
  - Справа/под ней rich‑text контент.
  - Внизу кнопка (если задана).
  - Крестик закрытия в углу.
- Стили:
  - Tailwind + shadcn‑диалог/Sheet или свой модальный компонент, в едином стиле с сайтом.
  - На мобильных — почти fullscreen sheet.

### 5. Границы первой итерации

Включаем:

- Один глобальный попап только для главной страницы.
- Полностью ручной rich‑text контент + одна картинка с d&d upload.
- Таймер показа, ограничение частоты (`hideForDays`), опциональное автозакрытие.
- Опциональная кнопка/ссылка (можно указывать `/catalog/aktsii`).

Откладываем на потом:

- Таргетинг по другим страницам/категориям.
- Несколько разных попапов и приоритеты.
- A/B‑тесты попапов.
- Сложные сценарии аналитики по показам/кликам (пока достаточно базовой аналитики через существующий трекинг, если нужно).

Этот документ описывает согласованный дизайн попап‑редактора. Следующий шаг — написать план реализации и подключить его к текущей архитектуре (Prisma, админка, фронт главной страницы).

# Доставляемость писем (без учёта DNS/домена)

Кратко: задержка между отправками и прочие меры, которые можно реализовать в коде и контенте, без настройки SPF/DKIM/DMARC.

## Задержка между отправками

Между письмом админам и письмом заказчику вставляется случайная задержка, чтобы не отправлять два письма подряд одним burst'ом.

- **По умолчанию:** 5–15 секунд (случайное значение в этом диапазоне).
- **Настройка:** в `.env` задать (значения в миллисекундах):
  - `EMAIL_SEND_DELAY_MS_MIN` — минимум задержки (например `10000` = 10 с).
  - `EMAIL_SEND_DELAY_MS_MAX` — максимум (например `60000` = 1 мин; верхний предел в коде — 25 мин).

Для 1–2 писем на заказ практика обычно достаточна 5–15 с; при большом потоке заказов можно поднять до 1–5 мин.

## Другие меры (уменьшение вероятности попадания в спам)

Ниже — варианты, которые стоит рассмотреть помимо доменных записей (SPF, DKIM, DMARC и т.д.):

1. **Контент и разметка**
   - Соотношение текст/HTML: не только HTML, а и текстовая версия письма (у нас уже есть для заказа).
   - Избегать типичных спам-слов и избыточных заглавных, множественных восклицаний, «КЛИКНИ СЕЙЧАС!!!».
   - Не перегружать письмо одними ссылками; ссылки — по делу, с нормальным текстом вокруг.

2. **Заголовки**
   - `List-Unsubscribe` (если рассылка не транзакционная) — часть провайдеров это учитывает.
   - `Message-ID`, `Date`, `From`/`Reply-To` — корректные и консистентные (один и тот же домен/адрес).
   - `Reply-To` на реальный ящик поддержки, если не хотите получать ответы на noreply.

3. **Поведение и объём**
   - Не слать много писем одному адресату в короткий промежуток (у нас задержка между разными типами писем по заказу уже есть).
   - Ограничение частоты на одного пользователя (rate limit по email), если появятся сценарии с частыми письмами.

4. **Персонализация и релевантность**
   - Обращение по имени, понятный контекст («заказ №…», «благодарим за заказ») — уже делаем.
   - Тема и первый абзац без обманчивых или «рассылочных» формулировок.

5. **Отказ от рассылки**
   - Для маркетинговых писем — явная ссылка «Отписаться» и обработка отписок; для транзакционных (заказ, сброс пароля) обычно не требуется.

6. **Стабильность отправителя**
   - Один и тот же домен и адрес From (например support@innerhealth.ru) для всех транзакционных писем — уже так и сделано.

Реализация пунктов 1, 4 и 6 уже частично есть; при добавлении новых типов писем стоит сверяться с этим списком.

## Что пока не имплементировано

| Мера | Статус |
|------|--------|
| **Задержка между отправками** | ✅ Реализовано (5–15 с, настраивается через env). |
| **Текст + HTML** | ✅ У писем заказа, сброса пароля, верификации и т.д. есть и `text`, и `html`. |
| **Персонализация (имя, контекст)** | ✅ В письме заказчику — имя и состав заказа. |
| **Единый From** | ✅ Все письма идут с `SUPPORT_EMAIL_FROM` (support@…), включая сброс пароля. |
| **Reply-To** | ✅ Во всех письмах задаётся `replyTo` (по умолчанию тот же адрес; переопределяется через `SUPPORT_EMAIL_REPLY_TO`). |
| **Message-ID / Date** | ✅ Обычно подставляются nodemailer автоматически; явно не задаём. |
| **List-Unsubscribe** | ➖ Не требуется для текущих писем (все транзакционные). Нужен только для маркетинговых рассылок. |
| **Rate limit по email** | ❌ Нет отдельного лимита «не более N писем на этот email в час». Есть только rate limit по IP на создание заказа. Имеет смысл при появлении сценариев с частой отправкой одному адресату. |
| **Отписка (маркетинг)** | ➖ Не применимо: только транзакционные письма. |

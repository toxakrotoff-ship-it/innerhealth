## 1. Секретный путь админки (ADMIN_SECRET_PATH)

**Проблема:** В `authorized` используется `ADMIN_SECRET_PATH`, но `config.matcher` зашит под `/admin/:path*`. При смене переменной защищённый путь и маршруты расходятся.

**Подход:** Один конфиг-префикс для админки. Маршруты приложения остаются под `/admin` (файловая структура не меняется). При запросе по кастомному URL — rewrite в middleware на `/admin/...`, чтобы Next отдавал те же страницы. Все ссылки в админке строятся от одного базового пути, который приходит с сервера.

**Детали:**

- **Переменная:** `ADMIN_SECRET_PATH` (без префикса `NEXT_PUBLIC_`). Значение по умолчанию: `'admin'`.
- **Middleware:**
  - `matcher`: строить динамически при загрузке модуля: `[ \`/${process.env.ADMIN_SECRET_PATH || 'admin'}/:path*\`, '/admin/:path*', '/api/admin/:path*', ... ]` — чтобы срабатывал и кастомный путь, и `/admin`.
  - В теле middleware: если `pathname.startsWith(\`/\${adminSecretPath}\`)` и `adminSecretPath !== 'admin'`, выполнить `NextResponse.rewrite(new URL(\`/admin\${pathname.slice(adminSecretPath.length + 1) || ''}\`, request.url))`, затем `next()`. Иначе — просто `next()`.
  - `authorized` без изменений: уже использует `adminSecretPath` для проверки.
- **Базовый путь для ссылок:** В админке все `href` должны вести на один и тот же префикс (тот, по которому пользователь зашёл). Поскольку после rewrite запрос приходит как `/admin/...`, в RSC мы не видим исходный URL. Поэтому базовый путь для ссылок задаём явно: `process.env.ADMIN_SECRET_PATH || 'admin'`. Все ссылки в админке формировать как `\${basePath}/catalog`, `\${basePath}/products/...` и т.д.
- **Передача basePath в клиент:** В `app/admin/layout.tsx` (RSC) вычислить `const adminBasePath = process.env.ADMIN_SECRET_PATH || 'admin'` и передать в `AdminLayoutClient` пропом `adminBasePath`. В `AdminLayoutClient` пробросить в `AdminNav` и везде, где есть ссылки на разделы админки. В `AdminNav` использовать `adminBasePath` для всех пунктов меню. Аналогично: кнопки «Назад к каталогу», ссылки «Ред.», «Добавить товар» и т.д. — везде подставлять `adminBasePath` (через контекст React или пропы).
- **Риск:** В Next.js `config.matcher` оценивается при сборке/загрузке; если `ADMIN_SECRET_PATH` задаётся только в рантайме (например, только на сервере), матчер может не включить кастомный путь. Решение: документировать, что переменная должна быть доступна при старте приложения (например, в `.env` на сервере).

**Файлы:** `src/proxy.ts`, `src/app/admin/layout.tsx`, `src/app/admin/components/AdminLayoutClient.tsx`, `src/app/admin/components/AdminNav.tsx`, страницы/компоненты админки с жёстко заданными `/admin/...` (каталог, товары, новости, заказы, промокоды, профиль, настройки и т.д.). Либо ввести один контекст `AdminBasePathContext` и хук `useAdminBasePath()` для единой точки подстановки.

## 2. SEO в форме товара

**Проблема:** В схеме и API есть `seoTitle`, `seoDescr`, `seoKeywords`; в форме создания/редактирования товара отдельного блока SEO нет.

**Подход:** Добавить в форму товара (новая и редактирование) секцию «SEO» с тремя полями; при сохранении отправлять их в API. API уже принимает эти поля в PUT/POST.

**Детали:**

- В `EditProductForm` и в форме создания товара (`products/new`): добавить в `formData` поля `seoTitle`, `seoDescr`, `seoKeywords`; при загрузке товара/инициализации — заполнять из `data`; в JSX — блок с подписью «SEO» и тремя полями (input для title/keywords, textarea для description).
- В `handleSubmit` (PUT) и в создании (POST) включать в body `seoTitle`, `seoDescr`, `seoKeywords` (уже в whitelist полей в API).

**Файлы:** `src/app/admin/products/[id]/edit/EditProductForm.tsx`, `src/app/admin/products/new/page.tsx` (или общий компонент формы, если вынесен).

## 3. Инлайн-редактирование цены и остатка в таблице каталога

**Проблема:** ТЗ требует редактирование цены и остатка прямо в таблице; сейчас только переход на страницу редактирования.

**Подход:** В таблице каталога (`ProductTable`) ячейки «Цена» и «Остаток» (quantity) сделать редактируемыми: по клику/фокусу — input, по blur или Enter — сохранение через API и обновление локального состояния. Один PATCH-эндпоинт для частичного обновления полей товара минимизирует нагрузку.

**Детали:**

- **API:** Добавить `PATCH /api/admin/products` (или расширить существующий роут): body `{ id, price?: number, quantity?: number }`. В хендлере — проверка сессии, затем `prisma.product.update({ where: { id }, data: { price?, quantity? } })`. Валидация: id обязателен, числа — не отрицательные.
- **Таблица:** В `ProductTable` в текущей разметке есть колонки «Цена» и (если есть) «Остаток». Сейчас колонки «Дата» и «Действия». Добавить колонку «Остаток» (quantity), если её нет. Для «Цена» и «Остаток»: по умолчанию показывать текст; при двойном клике (или кнопка «Изм.») переключаться в режим редактирования (input). По blur или Enter — вызвать PATCH, при успехе обновить `products` через callback `onRefresh()` (или точечно обновить элемент в state родителя). Отображать индикатор загрузки/ошибки (тост или inline).
- **Ссылки:** Использовать `adminBasePath` для ссылки «Ред.» (переход на страницу редактирования), чтобы после внедрения п.1 всё продолжало работать.

**Файлы:** `src/app/api/admin/products/route.ts` (PATCH), `src/app/admin/catalog/components/ProductTable.tsx`, при необходимости — мелкие правки в типе Product (quantity уже есть в схеме).

## 4. Галерея изображений с Drag-and-Drop сортировкой

**Проблема:** ТЗ — галерея с D&D; сейчас одно поле «Фото (URL)».

**Подход:** Хранить несколько фото в виде упорядоченного списка. Первое фото считаем основным (для карточки и превью). В БД — новое поле `photos` (Json, массив строк URL/путей). Поле `photo` оставить и синхронизировать с первым элементом `photos` при сохранении/чтении для обратной совместимости.

**Детали:**

- **Схема Prisma:** Добавить к модели `Product` поле `photos Json?` (массив строк). Опционально: при миграции заполнить `photos = [product.photo]` где `photo` не null. При чтении: если `photos` есть и массив не пустой, использовать его; иначе fallback на `[photo]`.
- **Форма товара:** Блок «Галерея»: список превью с возможностью добавить URL (или загрузка файла через существующий upload), удалить, перетащить для смены порядка. Библиотека: `@dnd-kit/core` + `@dnd-kit/sortable` (лёгкие и доступные) или аналог. Сохранение: в API передавать `photos: string[]` и при сохранении записывать первый элемент в `photo` для обратной совместимости.
- **API:** В PUT/POST добавлять приём `photos` (массив строк); валидация — массив, элементы — строки; запись в `product.photos` и синхронизация `photo = photos?.[0] ?? null`.
- **Публичная карточка товара:** Показывать галерею из `photos` (или fallback на `photo`), главное фото — первое в списке.

**Файлы:** `prisma/schema.prisma`, миграция, `src/app/api/admin/products/route.ts`, `src/app/admin/products/[id]/edit/EditProductForm.tsx`, форма создания товара, компонент галереи (новый), страница карточки товара (если показ галереи там нужен).

## 5. Правила промокодов (Rule A/B/C) и корзина/оформление

**Проблема:** По ТЗ: Rule A — если у товара уже есть скидка (`discountPrice != null`), промокод не применяется; Rule B/C — промокод только если `isPromoEligible === true` и нет `discountPrice`. В схеме Product нет полей `discountPrice` и `isPromoEligible`.

**Подход:** Ввести в Product поля `discountPrice` и `isPromoEligible`; в корзине и при создании заказа считать «регулярную» сумму только по позициям, где нет акционной цены и товар участвует в промо; скидку по промокоду применять только к этой сумме.

**Детали:**

- **Схема:** В модель `Product` добавить `discountPrice Float?` (акционная цена, если задана — считаем товар «уже со скидкой»), `isPromoEligible Boolean @default(true)` (участвует ли в промо). Миграция.
- **Админка, форма товара:** В форме редактирования/создания добавить поля «Акционная цена» (`discountPrice`) и чекбокс «Участвует в промокодах» (`isPromoEligible`). API PUT/POST уже можно расширить списком полей; при сохранении передавать эти два поля.
- **Правило в коде:** Товар «со скидкой» для целей промо: `discountPrice != null` ИЛИ (как сейчас) `priceOld != null && priceOld > price`. База для скидки по промокоду: только строки, где товар не «со скидкой» и `isPromoEligible === true`. Остальные строки в итоговую сумму входят без скидки по промокоду.
- **Корзина (клиент):** В `CartLine` добавить `isPromoEligible?: boolean`. При добавлении в корзину (страница товара, карточка) передавать с сервера/API `price`, `priceOld`, `discountPrice`, `isPromoEligible`. Вычислять `hasPromoPrice` как «уже со скидкой» (discountPrice задан или priceOld > price). В расчёте `subtotalRegular`: только позиции, где `!hasPromoPrice && isPromoEligible`; остальные — в «акционную» часть без скидки по промокоду.
- **Заказ (API):** В `POST /api/orders` при расчёте сумм по строкам использовать те же правила: выборка продуктов с полями `price`, `priceOld`, `discountPrice`, `isPromoEligible`; разделение на «со скидкой» и «регулярные, участвующие в промо»; применение промокода только к сумме вторых.
- **Валидация промокода:** `POST /api/promo/validate` без изменений по правилам A/B/C (правила применяются на этапе расчёта корзины и заказа, а не при валидации кода).

**Файлы:** `prisma/schema.prisma`, миграция, `src/app/api/admin/products/route.ts`, форма товара (edit/new), `src/store/cart-store.ts`, `src/components/site/cart-page-content.tsx`, места добавления в корзину (передача isPromoEligible и флага «со скидкой»), `src/app/api/orders/route.ts`, типы/интерфейсы CartLine и ответов API товара.

## 6. NextAuth v5 и 2FA

**Проблема:** Сейчас NextAuth v4; ТЗ — v5 и 2FA.

**Подход:** Поэтапно: (1) обновить NextAuth до v5 (Auth.js), сохранив Credentials + JWT и сброс пароля; (2) добавить 2FA по TOTP для ролей admin/writer (опционально или обязательно — настраиваемо).

**Детали:**

- **NextAuth v5:** Следовать официальной инструкции миграции (v4 → v5): новый пакет `next-auth@beta` или стабильный v5, структура `auth.config.ts`, `auth.ts`, роут в `app/api/auth/[...nextauth]/route.ts`, использование `getServerSession` из новой точки входа. Credentials provider и JWT callbacks перенести; проверить middleware (withAuth и т.д.) на совместимость с v5.
- **2FA (TOTP):** В модели User добавить поля `totpSecretEncrypted String?`, `totpEnabled Boolean @default(false)`. При первом включении 2FA: генерировать секрет, показывать QR и коды восстановления, сохранять зашифрованный секрет. При логине: после успешного пароля, если у пользователя `totpEnabled`, требовать одноразовый код; проверка через `otplib` (или аналог). Страница профиля в админке: блок «Двухфакторная аутентификация» — включить/выключить, смена секрета. Резервные коды хранить хэшированными или выдавать один раз и не хранить — на выбор по безопасности.
- **Обратная совместимость:** Пользователи без 2FA продолжают логиниться как раньше. После миграции все сессии могут сброситься — предупредить в релизе.

**Файлы:** `package.json`, `src/lib/auth.ts` (или разбить на auth.config.ts + auth.ts), `src/app/api/auth/[...nextauth]/route.ts`, middleware (при необходимости), страница логина (форма + шаг 2FA), страница профиля админки (2FA настройки), модель User и миграция (totpSecretEncrypted, totpEnabled), скрипты/утилиты для TOTP.

## Порядок реализации и зависимости

1. **Секретный путь** — независимый; можно делать первым.
2. **SEO в форме** — независимый.
3. **Инлайн-редактирование** — зависит от п.1 только в части ссылок (basePath).
4. **Галерея D&D** — миграция и форма; можно после 2.
5. **Промокоды** — миграция Product, затем форма, корзина, заказы.
6. **NextAuth v5 + 2FA** — последний, т.к. затрагивает всё приложение и сессии.

Рекомендуемый порядок выполнения в коде: 1 → 2 → 3 → 4 → 5 → 6.

## Критерии приёмки (кратко)

- При `ADMIN_SECRET_PATH=secret-panel` админка открывается по `/secret-panel/...`, все ссылки ведут на `/secret-panel/...`, API и страницы те же.
- В форме товара есть блок SEO; значения сохраняются и отображаются в просмотре.
- В таблице каталога можно менять цену и остаток инлайном; после сохранения список обновляется.
- У товара есть галерея (список фото с D&D); первое фото — основное; на карточке товара отображается галерея.
- В Product есть discountPrice и isPromoEligible; в корзине и заказе скидка по промокоду применяется только к подходящим товарам (Rules A/B/C).
- Приложение работает на NextAuth v5; для админов доступно включение 2FA по TOTP и вход по коду после пароля.

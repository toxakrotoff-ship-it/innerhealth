# 2FA: почта или Google Authenticator

**Дата:** 2026-02-24  
**Цель:** Двухфакторная аутентификация для входа в админку с выбором способа — код по email или TOTP (Google Authenticator и аналоги). Схема закладывается на User без привязки к роли, чтобы позже можно было включить 2FA для других типов пользователей.

**Подход:** Двухшаговый логин: проверка пароля → при включённом 2FA выдача одноразового «ожидания 2FA» → ввод кода → одноразовый grant → создание сессии через существующий Credentials provider.

---

## 1. Модель данных

### User (дополнительно)

| Поле | Тип | Описание |
|------|-----|----------|
| `twoFactorEnabled` | Boolean @default(false) | Включён ли 2FA для пользователя. |
| `twoFactorMethod` | String? | `'email'` \| `'totp'` — способ 2FA. null пока 2FA выключен. |
| `totpSecretEncrypted` | String? | Секрет TOTP, зашифрованный (например AES с ключом из env). Только при method = totp. |

Связи: без новых связей на User для 2FA (токены в отдельных таблицах).

### TwoFactorPending (новая таблица)

Хранит состояние «пароль проверен, ждём ввод кода». Жизнь — 5–10 минут, одна запись на пользователя (или перезаписываем по новому логину).

| Поле | Тип | Описание |
|------|-----|----------|
| id | String @id | cuid. |
| userId | String | Ссылка на User. |
| tokenHash | String | bcrypt-хеш токена, который отдаём клиенту (в cookie httpOnly). |
| expiresAt | DateTime | Удалять/игнорировать после. |
| createdAt | DateTime | Для аудита. |

Индексы: userId (уникальный — один pending на пользователя), expiresAt (очистка просроченных).

### TwoFactorGrant (новая таблица)

Одноразовый grant для завершения входа после успешной проверки 2FA.

| Поле | Тип | Описание |
|------|-----|----------|
| id | String @id | cuid (или случайный opaque token — по нему ищем). |
| userId | String | Ссылка на User. |
| usedAt | DateTime? | null = не использован; после использования — время. |
| expiresAt | DateTime | Короткий TTL (например 2 минуты). |
| createdAt | DateTime | Для аудита. |

Индексы: id (уникальный), userId, expiresAt. После использования grant не удаляем, а помечаем usedAt (аудит).

### Email-коды для 2FA (при method = email)

Вариант A — отдельная таблица `TwoFactorEmailCode`: userId, codeHash, expiresAt (5–10 мин).  
Вариант B — хранить последний отправленный код в TwoFactorPending: поле `emailCodeHash`, `emailCodeExpiresAt`; при запросе «отправить код» перезаписываем и шлём письмо.

Рекомендация: **Вариант B** — меньше таблиц, один pending на пользователя уже есть; при повторной отправке кода обновляем код и срок в той же записи.

Итого в **TwoFactorPending** добавляем (если method = email):

| Поле | Тип | Описание |
|------|-----|----------|
| emailCodeHash | String? | bcrypt-хеш 6-значного кода. |
| emailCodeExpiresAt | DateTime? | Срок действия кода (например 5 мин). |

## 2. Поток входа (логин)

1. **Форма логина** (как сейчас): email + пароль.
2. **Клиент** вызывает не `signIn` сразу, а `POST /api/auth/login-step1` с `{ email, password }`.
3. **login-step1**:
   - Проверяет email+пароль (та же логика, что в `authorize`).
   - Если неверно → 401.
   - Если верно и у пользователя `twoFactorEnabled === false` → 200 `{ success: true }`. Клиент тогда вызывает `signIn('credentials', { email, password })` и редирект как сейчас (change-password или /admin/catalog).
   - Если верно и 2FA включён → создаём/обновляем запись TwoFactorPending (tokenHash от случайного токена, expiresAt = now + 10 min). Если method = email — генерируем 6-значный код, пишем emailCodeHash, emailCodeExpiresAt, вызываем отправку письма (функция типа `send2FACodeEmail`). Устанавливаем httpOnly cookie `two_factor_pending` с подписанным/opaque токеном (или ставим в cookie только идентификатор pending-записи + подпись). Ответ 200 `{ need2FA: true, method: 'email' | 'totp' }`. Редирект на `/login/2fa`.
4. **Страница /login/2fa**:
   - Показывает поле ввода кода; при method = email — кнопка «Отправить код повторно» (rate limit: раз в 1–2 мин).
   - Отправка кода: `POST /api/auth/2fa/send-code` (читает cookie two_factor_pending, обновляет код в TwoFactorPending, шлёт письмо). При method = totp кнопки «отправить код» нет.
5. **Клиент** отправляет код в `POST /api/auth/verify-2fa` с телом `{ code: string }`. Cookie two_factor_pending отправляется автоматически.
6. **verify-2fa**:
   - По cookie находит TwoFactorPending, проверяет срок.
   - Для email: сверяет code с emailCodeHash (bcrypt).
   - Для totp: проверяет code через TOTP (speakeasy/otplib) по totpSecretEncrypted пользователя.
   - При неверном коде → 401, счётчик попыток при желании (опционально).
   - При верном: создаёт запись TwoFactorGrant (userId, expiresAt = now + 2 min), удаляет/инвалидирует TwoFactorPending, очищает cookie two_factor_pending, возвращает 200 `{ grantToken: string }` (opaque id записи Grant или подписанный JWT с grantId).
7. **Клиент** вызывает `signIn('credentials', { grantToken })`. В Credentials provider в `authorize`: если передан `grantToken`, ищем TwoFactorGrant по id/токену, проверяем usedAt === null и expiresAt > now, помечаем usedAt = now, загружаем user по userId и возвращаем объект user (как при обычном логине). Иначе — текущая логика email+password (но тогда при 2FA мы в step1 уже не отдаём success, так что с паролем без grant сюда не попадём при включённом 2FA; для пользователей без 2FA step1 отдаёт success и клиент шлёт email+password в signIn).
8. После успешного signIn — редирект на change-password или /admin/catalog, как сейчас.

Итого: сессия создаётся только после проверки 2FA; пароль по сети при 2FA не летит повторно (только при первом шаге).

## 3. Настройка 2FA (включение / смена способа / отключение)

Доступно только авторизованному пользователю (в админке, например раздел «Безопасность» или «Профиль»).

- **Включение 2FA:** пользователь выбирает способ (email или приложение).  
  - **Email:** сохраняем twoFactorMethod = 'email', twoFactorEnabled = true. Коды при логине будут отправляться на email пользователя (или notificationEmail, если задан — уточнить продуктово).  
  - **TOTP:** генерируем секрет (speakeasy), сохраняем зашифрованно в totpSecretEncrypted, показываем QR и manual key; пользователь вводит первый код для проверки → проверяем TOTP → twoFactorMethod = 'totp', twoFactorEnabled = true.
- **Смена способа:** по сути повтор включения (перегенерировать TOTP или переключить на email).
- **Отключение 2FA:** запрос с текущим паролем (или повторным 2FA-кодом) → twoFactorEnabled = false, twoFactorMethod = null, totpSecretEncrypted = null.

API: например `POST /api/auth/2fa/setup` с телом `{ action: 'enable' | 'disable', method?: 'email' | 'totp', code?: string, password?: string }` и т.п. Детали — в плане реализации.

## 4. Безопасность и обработка ошибок

- Пароль при 2FA не передаётся на втором шаге; в cookie только одноразовый pending-токен с коротким TTL.
- Grant одноразовый (usedAt), короткий TTL.
- Pending и grant храним/инвалидируем в БД; cookie подписывать (или хранить в cookie только id и проверять подпись на сервере).
- Email-коды: 6 цифр, срок жизни 5–10 мин; rate limit на отправку (например раз в 1 мин на пользователя).
- При множественных неудачных попытках 2FA можно временно блокировать (опционально в первой итерации — не обязательно).
- Логирование: успешный/неуспешный 2FA, использование grant (для аудита).

---

## 5. Что не входит в первую итерацию

- Обязательный 2FA по политике (флаг «требовать 2FA» для роли/всех) — можно добавить позже полем на User или глобальной настройкой.
- Восстановление доступа при потере телефона (recovery codes) — отдельная фича.
- 2FA для клиентских аккаунтов — схема уже готова (поля на User), включение в UI и логике позже.

---

## 6. Стек и зависимости

- **TOTP:** библиотека `otplib` или `speakeasy` (генерация секрета, проверка кода).  
- **Шифрование секрета TOTP:** например `crypto.createCipheriv`/`createDecipheriv` (AES-256-GCM) с ключом из env `TOTP_SECRET_ENCRYPTION_KEY` (32 байта base64).  
- **Email:** существующий `sendInitialPasswordCodeEmail`-подобный helper — добавить `send2FACodeEmail(to, code)`.  
- **Cookie:** подпись через тот же механизм, что и сессии (NEXTAUTH_SECRET или отдельный ключ).

Документ можно использовать как спецификацию для плана реализации (writing-plans).

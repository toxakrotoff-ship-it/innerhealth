# План: завершение регистрации нового пользователя (ссылка + код по почте)

**Статус:** реализовано. Актуальное состояние — [STATUS.md](../STATUS.md) (раздел «Пользователи», модель SetInitialPasswordToken, страница `/login/set-initial-password`). Ниже — исходное описание плана.

Пользователю при создании аккаунта админом не отправляется пароль. Вместо этого отправляется письмо со ссылкой; по переходу по ссылке отправляется второе письмо с 6-значным кодом; после ввода кода пользователь задаёт пароль (без ввода старого).

---

## 1. Общая схема

- **При создании пользователя (админ):** пароль в письме не отправляется. Создаётся одноразовая ссылка (токен в URL), пользователю уходит **письмо 1**: «Для завершения регистрации пройдите по: {ссылка}».
- **По переходу по ссылке:** форма пароля сразу не показывается. На почту уходит **письмо 2**: «Ваш одноразовый код: {6 цифр}». На странице показывается форма ввода кода.
- **После ввода кода:** показывается форма «Новый пароль»; по отправке пароль устанавливается (без старого). Ссылка и код одноразовые и с ограничением по времени.

Ссылка даёт право на установку пароля без старого; 6-значный код подтверждает доступ к почте в момент завершения.

## 2. Где что менять

### База и токены

- Сейчас для «забыл пароль» используется **`PasswordResetToken`** (id, userId, tokenHash, expiresAt, usedAt). Ссылка: `.../login/reset-password?token={recordId}.{secret}`.
- Для «завершение регистрации» лучше завести **отдельную сущность**, чтобы не смешивать с обычным сбросом и хранить код подтверждения. Варианты:
  - **Вариант A:** новая модель, например `SetInitialPasswordToken`: поля `id`, `userId`, `tokenHash`, `expiresAt`, `usedAt`, а также `emailCodeHash` (nullable), `emailCodeExpiresAt` (nullable), `codeVerifiedAt` (nullable). При первом открытии ссылки генерируется 6-значный код, пишется хэш и срок в эти поля, отправляется письмо 2.
  - **Вариант B:** та же таблица + поле `type: 'reset' | 'initial'` и поля для кода. Работает, но смешивает два сценария.

Рекомендация: отдельная модель (или отдельная таблица) под «initial set password» с полями для кода и его проверки.

### Письма

- В **`src/lib/email.ts`**:
  - Добавить функцию вида **`sendInitialPasswordLinkEmail(to, link)`** — текст: «Для завершения регистрации пройдите по: {link}» (и при необходимости HTML-версия).
  - Добавить функцию **`sendInitialPasswordCodeEmail(to, code)`** — текст: «Ваш одноразовый код: {code}» (6 цифр).

- При создании пользователя в **`/api/admin/users`** (POST) вместо `sendNewUserCredentials` вызывать создание токена «set initial password» + `sendInitialPasswordLinkEmail`.

### Создание пользователя (админ)

- **`nextjs-project/src/app/api/admin/users/route.ts`** (POST):
  - Не генерировать пароль для письма и не вызывать `sendNewUserCredentials`.
  - Создать пользователя с временным/случайным паролем (как сейчас) и сразу создать запись токена «set initial password» (аналог того, как в `forgot-password` создаётся `PasswordResetToken`): сгенерировать `secret`, посчитать `tokenHash`, сохранить в новой таблице с `userId` и `expiresAt` (например 24–48 часов).
  - Собрать ссылку: `{baseUrl}/login/set-initial-password?token={recordId}.{secret}` (как в forgot-password).
  - Отправить **только** письмо 1 через `sendInitialPasswordLinkEmail(user.email, link)`.
  - В ответе API можно вернуть что-то вроде `emailSent: true`, без пароля.

### Маршруты и страницы

- **Новый URL** для завершения регистрации, например: **`/login/set-initial-password`** (отдельно от `/login/reset-password`).
- **Страница** `src/app/login/set-initial-password/page.tsx`:
  - Читает `token` из query (как на reset-password).
  - **Шаг 1 (по умолчанию):** форма «Введите код из письма» (одно поле, 6 цифр).
  - При открытии страницы с валидным токеном: если код ещё не отправлялся — на бэкенде сгенерировать 6-значный код, сохранить хэш и срок, отправить письмо 2 (см. ниже). Если код уже отправлялся — не слать повторно (или лимит раз в N минут).
  - Отправка кода с формы: **POST** к новому API (например `/api/auth/verify-initial-code`) с `{ token, code }`. При успехе — перейти на шаг 2.
  - **Шаг 2:** форма «Новый пароль» + «Повторите пароль» (как на reset-password). Отправка: **POST** к новому API (например `/api/auth/set-initial-password`) с `{ token, password }`. При успехе — редирект на логин и сообщение «Пароль установлен».

### API для «завершение регистрации»

- **GET или POST «открытие ссылки» (отправка кода на почту):**  
  Лучше вызывать при загрузке страницы с валидным токеном (например отдельный endpoint типа `POST /api/auth/send-initial-password-code` с телом `{ token }`). Логика:
  - Разобрать `token` (recordId + secret), найти запись, проверить `tokenHash` и `expiresAt`, что `usedAt` пустой.
  - Если для этой записи ещё нет `emailCodeHash` (или код просрочен и политика разрешает переотправку) — сгенерировать 6-значный код, сохранить хэш и `emailCodeExpiresAt` (например 10–15 минут), отправить письмо 2 через `sendInitialPasswordCodeEmail(user.email, code)`.
  - Ответ: `{ ok: true }` или ошибка (неверный/просроченный токен, лимит отправки и т.д.).

- **POST `/api/auth/verify-initial-code`** (body: `token`, `code`):
  - Найти запись по token (id + secret), проверить срок и что не использована.
  - Проверить код через bcrypt.compare(code, record.emailCodeHash).
  - Если ок — выставить `codeVerifiedAt = now()`, вернуть успех. Front после этого показывает форму пароля.

- **POST `/api/auth/set-initial-password`** (body: `token`, `password`):
  - Снова найти и проверить запись по token; проверить, что есть `codeVerifiedAt` и нет `usedAt`.
  - Обновить пароль пользователя (hash + сохранить), выставить `usedAt` на записи токена.
  - Вернуть успех; front редирект на логин.

Итого: письмо 1 — только ссылка; по ссылке — отправка кода и форма кода; после проверки кода — форма пароля; пароль задаётся без старого, всё привязано к одноразовой ссылке с хэшем.

### Безопасность

- Токен в ссылке — криптостойкий (как в `password-reset`: id + secret, secret хранится в хэше). Код — 6 цифр, хранить в БД только хэш (bcrypt), срок жизни кода короткий (10–15 минут). Один токен — один раз использование (`usedAt`). Rate limit на отправку кода и на проверку кода/установку пароля (как в `forgot-password`/`reset-password`).

## 4. Удаление пользователей (admin-panel/users)

В админке «Пользователи» нужна кнопка удаления пользователя с подтверждением.

### Бэкенд

- **Файл:** `nextjs-project/src/app/api/admin/users/[id]/route.ts`.
- Добавить обработчик **DELETE**:
  - Проверка прав: только роль `ADMIN` (как в существующем PATCH).
  - Запретить удаление самого себя: если `session.user.id === id` — ответ 403 (например: «Нельзя удалить свой аккаунт»).
  - Связанные сущности: у `User` связи с `TelegramWhitelist`, `TelegramLinkCode`, `PasswordResetToken` идут с `onDelete: Cascade` — записи удалятся вместе с пользователем.
  - У модели `Order` есть поле `userId` (optional). Если в схеме нет обратной связи с `onDelete`, при наличии заказов у пользователя удаление может упереться в FK. В этом случае в DELETE сначала обнулить заказы: `prisma.order.updateMany({ where: { userId: id }, data: { userId: null } })`, затем `prisma.user.delete({ where: { id } })`.
  - Ответ: при успехе — 200 или 204; если пользователь не найден — 404; при попытке удалить себя — 403.

### Фронт

- **Файл:** `nextjs-project/src/app/admin/users/page.tsx`.
- Добавить колонку «Действия» (или иконку/кнопку в существующей строке таблицы).
- Кнопка «Удалить» (иконка корзины) у каждой строки.
- По клику — подтверждение: `window.confirm('Удалить пользователя …?')` или модальное окно (если в проекте есть UI для модалок).
- После подтверждения: `fetch(\`/api/admin/users/${user.id}\`, { method: 'DELETE', credentials: 'include' })`. При успехе — обновить список (`fetchUsers()`); при ошибке — показать сообщение из `data.error`.

### Безопасность

- Удалять может только ADMIN.
- Текущий пользователь не может удалить себя через этот интерфейс.

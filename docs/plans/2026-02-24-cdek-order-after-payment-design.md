# Создание заказа на отгрузку в СДЭК после оплаты

**Дата:** 2026-02-24  
**Цель:** Автоматически создавать заказ на отгрузку в СДЭК при переходе заказа в статус «Оплачен» — только для заказов с доставкой СДЭК (до ПВЗ или до двери). При ошибке — несколько повторов, затем флаг и кнопка в админке «Создать отгрузку в СДЭК».

**Документация СДЭК API:** [apidoc.cdek.ru](https://apidoc.cdek.ru/)

---

## 1. Модель данных (вариант 1 — явные поля)

### Order (дополнительно)

| Поле             | Тип     | Описание |
|------------------|--------|----------|
| `cdekOrderUuid`  | String? @unique | UUID заказа в СДЭК после успешного создания. |
| `cdekOrderError`  | String? | Последнее сообщение об ошибке СДЭК (если заказ не создан), для отображения в админке и повтора. |

### ShippingInfo (дополнительно)

**Тип доставки и данные СДЭК:**

| Поле             | Тип     | Описание |
|------------------|--------|----------|
| `deliveryMethod` | String? | `'cdek_pvz'` \| `'cdek_door'` \| null. Если null — доставка не СДЭК, заказ в СДЭК не создаём. |
| `cdekCityCode`   | Int?    | Код города СДЭК получателя. |
| `cdekPvzCode`    | String? | Код ПВЗ (только для `cdek_pvz`). |
| `cdekTariffCode` | Int?    | Код тарифа, выбранный при расчёте (136, 139, 234 и т.д.). |

**Структурированный адрес «до двери»** (используется при `deliveryMethod = 'cdek_door'`):

| Поле        | Тип    | Описание |
|-------------|--------|----------|
| `street`    | String? | Улица. |
| `house`     | String? | Дом. |
| `apartment` | String? | Квартира. |
| `entrance`  | String? | Подъезд. |
| `floor`     | String? | Этаж. |
| `intercom`  | String? | Домофон. |

Поле `address` сохраняем: для «до двери» заполняем склеенной строкой из структурированных полей (для писем и отображения); для ПВЗ — адрес ПВЗ, как сейчас.

---

## 2. Оформление заказа (корзина → API)

- В запросе POST `/api/orders` помимо текущего `shipping` передаём:
  - `deliveryMethod`: `'cdek_pvz'` \| `'cdek_door'` \| null (если не СДЭК — не передаём или null).
  - При СДЭК: `cdekCityCode`, `cdekTariffCode`, при ПВЗ — `cdekPvzCode`; при «до двери» — объект `doorAddress`: `{ street, house, apartment, entrance, floor, intercom }`.
- Валидация (Zod): расширить `shippingSchema` / body: опциональные `deliveryMethod`, `cdekCityCode`, `cdekPvzCode`, `cdekTariffCode`, `doorAddress` (объект с опциональными строковыми полями).
- В транзакции создания заказа при создании `ShippingInfo` записывать все новые поля; для «до двери» при наличии `doorAddress` сохранять структурированные поля и формировать `address` одной строкой для совместимости.

---

## 3. Webhook ЮKassa: создание заказа в СДЭК при оплате

- В `POST /api/webhooks/yookassa` при `event === 'payment.succeeded'` и переходе заказа в `paid`:
  1. Загрузить заказ с `shippingInfo` (включая `deliveryMethod`, `cdekCityCode`, `cdekPvzCode`, `cdekTariffCode`, структурированный адрес).
  2. Если `deliveryMethod` не `'cdek_pvz'` и не `'cdek_door'` — ничего не делать по СДЭК.
  3. Если уже есть `cdekOrderUuid` — не создавать повторно.
  4. Иначе вызвать сервис создания заказа в СДЭК (см. ниже). При успехе — обновить заказ: `cdekOrderUuid = <uuid>`, `cdekOrderError = null`. При ошибке — retry 2–3 раза с задержкой (например 2–5 сек); если всё равно ошибка — записать в `cdekOrderError` текст ошибки, заказ остаётся оплаченным.

---

## 4. Создание заказа в СДЭК (логика)

- Вынести в `src/lib/cdek.ts` (или отдельный модуль) функцию `createCdekOrder(orderId: string): Promise<{ uuid: string } | { error: string }>`.
- Внутри: загрузка заказа с items (product — габариты), shippingInfo, настройки отправителя. Сборка тела запроса по [документации СДЭК](https://apidoc.cdek.ru/): отправитель (из настроек), получатель (из ShippingInfo), локации, пакеты (из товаров через `productToCdekPackage` / `mergeCdekPackages`), тариф, номер заказа (наш `orderId` или внешний номер). Для «до двери» — адрес из полей `street`, `house`, `apartment` и т.д.; для ПВЗ — код ПВЗ в пункте назначения.
- Отправитель: данные из админки (Настройки → СДЭК): имя, телефон, адрес, код города отправления. При отсутствии — fallback на переменные окружения (например `CDEK_SENDER_*`). Список полей уточнить по API СДЭК (sender / from_location).
- Вызов: POST к endpoint создания заказа (v2), с Bearer-токеном от `getCdekToken()`. Ответ: uuid созданного заказа или ошибка.

---

## 5. Настройки админки: отправитель СДЭК

- В блоке «Доставка (СДЭК)» на странице настроек добавить поля: имя отправителя, телефон, адрес (или структурировано: город, улица, дом и т.д. — по требованиям API), код города отправления (если нужен отдельно от `CDEK_FROM_CITY_CODE`). Сохранять в `SiteSetting` с ключами вида `cdek_sender_*`.
- Документировать возможность переопределения через env (например `CDEK_SENDER_NAME`, `CDEK_SENDER_PHONE`, `CDEK_FROM_CITY_CODE` уже есть).

---

## 6. Админка: кнопка «Создать отгрузку в СДЭК»

- На странице заказа (или в списке заказов) для оплаченных заказов с `deliveryMethod` = `'cdek_pvz'` или `'cdek_door'` и при этом без `cdekOrderUuid` (и опционально с заполненным `cdekOrderError`) показывать кнопку «Создать отгрузку в СДЭК».
- По нажатию: вызов API (например POST `/api/admin/orders/[id]/cdek-shipment`), который вызывает ту же логику создания заказа в СДЭК. При успехе — обновить заказ (`cdekOrderUuid`, очистить `cdekOrderError`); при ошибке — обновить `cdekOrderError` и показать сообщение пользователю.

---

## 7. Обработка ошибок

- Повторы при вызове из webhook: 2–3 попытки с задержкой (2–5 сек). После исчерпания — только запись в `cdekOrderError`, без падения webhook (ЮKassa должна получить 200).
- Логирование: все вызовы СДЭК и ошибки логировать (orderId, ответ/исключение) для разбора.
- Админка: при наличии `cdekOrderError` отображать текст; кнопка «Создать отгрузку в СДЭК» даёт возможность повтора без повторной оплаты.

---

## 8. Итог по шагам реализации

1. Миграция Prisma: новые поля в `Order` и `ShippingInfo` (включая структурированный адрес).
2. Валидация и API заказа: приём и сохранение `deliveryMethod`, данных СДЭК, `doorAddress`.
3. Корзина: передача новых полей и структурированного адреса при оформлении.
4. Настройки: поля отправителя СДЭК в админке (+ env fallback).
5. `src/lib/cdek.ts`: функция создания заказа в СДЭК по API.
6. Webhook ЮKassa: после перевода в paid вызов создания СДЭК с retry и записью `cdekOrderUuid` / `cdekOrderError`.
7. Админка: отображение статуса СДЭК по заказу и кнопка «Создать отгрузку в СДЭК» + API для повтора.

После согласования этого дизайна следующий шаг — детальный план реализации (writing-plans).

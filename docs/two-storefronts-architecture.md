# Архитектура: один бэкенд, две витрины (Inner Health · Sprint Power)

**Статус документа:** целевая модель и принятые решения (обновлено по согласованию 2026-05-04).  
**Связанные материалы:** [STATUS.md](./STATUS.md), [storefront-copy-ownership.md](./backend-notes/storefront-copy-ownership.md), [README.md](../README.md).  
**План внедрения:** [plans/2026-05-04-second-brand-completion-plan.md](./plans/2026-05-04-second-brand-completion-plan.md).

---

## 1. Инварианты платформы

| Принцип | Решение |
|--------|---------|
| Бэкенд | Один деплой Next.js + одна БД PostgreSQL. |
| Витрины | Две изолированные клиентские витрины (разные домены / канонические URL), общий подход к UI с **разным оформлением** (на старте — преимущественно **перекрас** через токены/тему; позже возможна более глубокая кастомизация). |
| Каталог и заказы | Товары и заказы **разведены по бренду** (товар принадлежит витрине бренда; смешанной корзины **нет**). |
| Настройки | Часть ключей **глобальные**, часть **brand-scoped** в админке (переключатель активного бренда). Конкретные ключи — в коде `SiteSetting` и доках env. |

---

## 2. Учётки клиентов и ЛК

- **Единая сеть учёток:** один пользователь — одна запись `User` (email глобально уникален). ЛК **объединённый**: заказы с обеих витрин доступны в одном кабинете (с понятной маркировкой бренда в UI — на усмотрение реализации).
- **Регистрация на второй витрине:** если email уже есть в системе, не создаём второй аккаунт; показываем смысл сообщения вроде: *«У вас уже есть аккаунт в нашей сети магазинов — войти?»* (точная формулировка — копирайт / content blocks).
- **Вход и регистрация:** **общие** маршруты (`/login`, `/register` и т.д.) для обеих витрин; бренд витрины определяется как сейчас (host / cookie / header).

---

## 3. Админка

- Команда админов **небольшая**; у всех админов **одинаковые полномочия** (без ролевого разделения «только Inner / только Sprint» на уровне продукта).
- Повседневная работа — в контексте **переключателя бренда**; списки заказов, каталог, контент по умолчанию в рамках выбранного бренда.

---

## 4. Партнёры

- Допускается сценарий **один партнёр — два бренда** (отдельные привязки промокодов/комиссий по бренду, в модели данных уже есть `brand` у сущностей партнёрских промокодов — см. Prisma).

---

## 5. Корзина и заказы

- **Смешанной корзины нет** (товары только одного бренда в одном оформлении).
- **Явный бренд заказа:** для отчётности, выгрузок и интеграций на сущности заказа — поле **`Order.brand`** (`inner` | `sprint-power`), выставляется при оформлении и используется в админке/синках вместо эвристики по строкам. Срез «как было до колонки»: [order-brand-phase-0-snapshot.md](./backend-notes/order-brand-phase-0-snapshot.md).

---

## 6. Платежи (ЮKassa)

- **Разные shopId** = разные магазины в ЮKassa. В админке при переключателе **Inner Health** / **Sprint Power** сохраняются отдельные `yookassa_shop_id`, `yookassa_secret_key`, `yookassa_term_id` и коды НДС для чеков (в БД — ключи с префиксом бренда, см. `YOOKASSA_BRAND_SCOPED_SETTING_KEYS` в `nextjs-project/src/lib/brand/brand-settings.ts`). Пустое значение у бренда — **fallback** на вкладку **Global**.
- **Env** `YOOKASSA_*` — запасной контур, если для контекста в настройках не задана пара shop id + secret (см. проверку подключения и оформление заказа).

---

## 7. Доставка (СДЭК)

- **Сейчас:** OAuth client id/secret и параметры отправителя читаются из **настроек админки** (`SiteSetting`). Для второго договора сохраните значения **в контексте бренда** (переключатель витрины / `?brand=sprint-power` в API настроек): в БД они лежат как `sprint-power:cdek_api_key` и т.д., с **fallback на глобальные** ключи без префикса, если для бренда пусто.
- **При создании заказа в СДЭК** используется `order.brand`: и учётные данные, и отправитель (см. `getCdekCredentials`, `resolveCdekSenderSettings` в `nextjs-project/src/lib/cdek.ts` и `CDEK_BRAND_SCOPED_SETTING_KEYS` в `brand-settings.ts`).

---

## 8. Боты (Telegram, Max и т.п.)

- **Разные боты** (и токены) на каждый бренд; настройки — brand-scoped там, где уже принято в коде.

---

## 9. Почта (SMTP)

- **На старте:** один и тот же SMTP-сервер и базовый адрес отправителя; при необходимости позже — разные From/шаблоны по бренду без смены «движка» отправки.

---

## 10. Контент и юридические страницы

- Юридические и статические информационные страницы должны **редактироваться из админки** (или через content blocks / выделенные поля настроек) с **разным текстом по бренду**, где это требуется законодательно или коммерчески.
- **В репозитории:** политика (`/privacy`) и оферта (`/oferta`) — страницы контент-блоков `legal-privacy` и `legal-oferta` (ключ `*.body`, rich), значения по бренду задаются переключателем бренда в админке; до заполнения показывается кодовый fallback (см. [storefront-copy-ownership.md](./backend-notes/storefront-copy-ownership.md)).
- **SEO:** два канонических домена; **избегать дублей** одинакового контента на обоих доменах (разные тексты/структура или noindex там, где копипаст неизбежен — политика задаётся владельцем продукта).

---

## 11. Визуал витрины

- **Текущая цель:** общий каркас + **перекрас** (CSS-переменные / тема бренда). На `<html>` выставляется `data-brand="inner" | "sprint-power"` (см. корневой `layout.tsx`); для Sprint переопределяются токены акцента (`--color-action-blue` и др. в `globals.css`), фон `body` и превью Open Graph отличаются от Inner.
- Более глубокое различие макетов — возможное будущее развитие, не блокирует текущую модель.

---

## 12. Отчётность и экспорт

- **По умолчанию:** фильтрация и выгрузки **в контексте одного бренда** (как в переключателе админки).
- **Дополнительно:** нужна возможность **выгрузить данные с обоих брендов** одной операцией (например, кнопка «с обеих витрин») для сводной отчётности на стороне пользователя (Excel/CSV и т.д.) — детали формата в задаче на реализацию.

---

## 13. Поддержка

- **Один пул** сотрудников поддержки с доступом к обоим брендам — целевая операционная модель.

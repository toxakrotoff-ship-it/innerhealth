# SOCKS5 к Telegram API в продакшене

Краткий гайд: как включить прокси для исходящих запросов к `api.telegram.org` с VPS (когда прямой доступ к Telegram заблокирован или нестабилен). Один и тот же формат ссылки, что в клиенте Telegram («Поделиться прокси» / ссылка вида `https://t.me/socks?...`).

Связанные документы: [tg_bot.md](./tg_bot.md), [deploy-vps.md](./deploy-vps.md).

---

## 1. Что настраивается

| Компонент | Зачем прокси |
|-----------|----------------|
| Контейнер **`app`** (Next.js) | Уведомления о заказах, формах, отзывах, ошибках ЮKassa и т.д. — все вызовы `sendMessage` / `getMe` идут через прокси, если переменная задана. |
| Контейнер **`telegram-bot`** (и при необходимости **`telegram-bot-sprint`**) | Long polling и ответы бота к Telegram API. |

Оба сервиса в `docker-compose.yml` используют **`env_file: .env`**. Достаточно **одной** строки в файле `.env` на сервере в каталоге **`nextjs-project/`** (рядом с `docker-compose.yml`).

---

## 2. Файл `.env` на проде

Откройте на сервере:

```bash
cd /opt/innerhealth/nextjs-project   # или ваш путь к проекту
nano .env
```

Добавьте **одну** переменную (имя можно использовать любое из пары):

```bash
TELEGRAM_SOCKS_PROXY_URL="https://t.me/socks?server=ВАШ_HOST&port=ВАШ_ПОРТ&user=ЛОГИН&pass=ПАРОЛЬ"
```

Либо альтернативное имя (эквивалент):

```bash
TELEGRAM_PROXY_SOCKS_URL="https://t.me/socks?server=..."
```

**Рекомендации:**

- Строку целиком возьмите из Telegram («Применить настройки прокси» / скопированная ссылка). Вставляйте в **двойных кавычках**, чтобы спецсимволы в пароле не сломали shell.
- Поддерживаются те же форматы, что парсит приложение: `https://t.me/socks?...`, `tg://socks?...`, `socks5://user:pass@host:port`. См. `nextjs-project/.env.example`.
- Файл `.env` **не коммитьте** в git; пароли храните только на сервере.

---

## 3. Перезапуск сервисов

После сохранения `.env` перезапустите процессы, которые ходят в Telegram API:

```bash
cd /opt/innerhealth/nextjs-project

docker compose up -d --force-recreate app telegram-bot
```

Если поднят второй бренд (Sprint Power):

```bash
docker compose up -d --force-recreate app telegram-bot telegram-bot-sprint
```

Пересборка образа **не обязательна**, если менялись только переменные окружения — достаточно recreate контейнеров, чтобы подтянуть новый `.env`.

---

## 4. Проверка

1. **Логи приложения** после тестового действия (например отправка формы или тестовый заказ): не должно быть стабильных ошибок `[telegram-notify] sendMessage failed` с таймаутом или `ECONNREFUSED` к `api.telegram.org`.
2. **Логи бота:** `docker compose logs -f telegram-bot` — после перезапуска не должно быть бесконечных ошибок при `getUpdates` / отправке сообщений.
3. В админке / ЛК сценарий **«Подключить Telegram»**: если `getMe` ходит через прокси, генерация ссылки на бота должна работать при корректном токене бота в настройках.

Если прокси с ошибкой в пароле или недоступен хост — в логах будет сообщение вида  
`[telegram-api-fetch] Invalid TELEGRAM_SOCKS_PROXY_URL` или сетевые ошибки при обращении к прокси.

---

## 5. Частые ошибки

| Симптом | Что проверить |
|---------|----------------|
| Уведомления с сайта не приходят, бот «живой» | Раньше прокси был только у бота: убедитесь, что **`TELEGRAM_SOCKS_PROXY_URL` в `.env` читает и `app`**. После правок `.env` сделайте `docker compose up -d --force-recreate app`. |
| Пароль со спецсимволами | Оставьте значение в **кавычках** в `.env`. |
| Два бота (inner / sprint-power) | Один и тот же SOCKS часто подходит для обоих; переменная общая. Если у ботов разные токены — это нормально, прокси один. |
| Ссылка из браузера обрезана | Копируйте полную строку `https://t.me/socks?server=...` без обрезки query-параметров. |

---

## 6. Безопасность

- Не публикуйте полную ссылку с `user` и `pass` в чатах и тикетах.
- При компрометации смените учётные данные прокси у провайдера / в Telegram и обновите строку в `.env`.

---
